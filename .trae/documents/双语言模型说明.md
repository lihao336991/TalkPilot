# TalkPilot 双语言模型说明

## 1. 结论

TalkPilot 当前产品语言模型只有两个核心概念：

- `母语 / Native language`
- `学习语言 / Learning language`

不要把当前产品理解成“三语言系统”。

尤其不要再把 `suggest`、`review`、`assist` 统一抽象成一个单独的 `helpLocale`。

## 2. 用户视角的定义

### 2.1 母语

母语用于：

- App 界面文案
- 解释类反馈
- 救场输入语言
- 把学习语言翻回用户能立刻理解的语言

### 2.2 学习语言

学习语言用于：

- 主练习链路
- 主 websocket 的实时识别语言
- `suggest` 输出内容
- `review` 中 corrected / better expression 的目标语言
- `assist` 最终翻译结果
- `assist` 的 TTS 朗读语言

## 3. 当前代码字段映射

当前代码里已经收敛为下面这套映射：

- `uiLocale`
  - 当前承载“母语文案”
  - 也就是当前产品意义上的 `nativeLanguage`
- `learningLanguage`
  - 当前承载“学习语言”

注意：

- 现在没有必要把当前产品再抽成第三个单独的 `helpLocale`
- `helpLocale` 已不再作为当前主业务模型使用

## 4. 设置页语义

设置页当前提供两个选项：

- `App language`
- `Learning language`

它们在产品上的真实含义分别是：

- `App language`
  - 当前近似等于 `母语`
  - 控制界面文案和解释语言
- `Learning language`
  - 当前等于 `学习语言`
  - 控制主练习链路的目标语言

## 5. 各链路的正确语言职责

### 5.1 主 websocket

- 监听 `学习语言`
- 这是主练习链路
- 例如用户学英语，主 WS 就监听英语
- 以后用户学日语，主 WS 就应该监听日语

### 5.2 Assist websocket

- 监听 `母语`
- 这条链路只用于救场
- 不是第二条主练习链路

### 5.3 Suggest

`suggest` 的职责是：

- 给用户一个“可以直接说出口”的目标句子

所以它应当表现为：

- `母语文案`
- `学习语言内容`

更具体地说：

- 标签、按钮、说明、容器文案走 `母语`
- suggestion 正文走 `学习语言`

### 5.4 Review

`review` 的职责是：

- 帮助用户理解自己说得哪里不对

所以它应当表现为：

- `母语文案`
- `学习语言内容`

更具体地说：

- issue explanation 走 `母语`
- praise 走 `母语`
- corrected / better expression 走 `学习语言`
- UI 标签走 `母语`

### 5.5 Assist 救场

`assist` 的标准链路应该是：

1. 母语实时识别
2. 拿到母语 transcript
3. 翻译成学习语言
4. 返回学习语言文本
5. 用学习语言做 TTS

## 6. 当前已落地实现

### 6.1 设置与状态

- 设置页位于 `app/settings.tsx`
- 页面实现位于 `src/features/settings/screens/SettingsScreen.tsx`
- 本地状态持久化位于 `src/shared/store/localeStore.ts`
- 当前状态字段：
  - `uiLocale`
  - `learningLanguage`

### 6.2 主链路与救场链路

- `DeepgramStreamingService`
  - 已按 `learningLanguage` 建连
- `AssistStreamingService`
  - 已按 `uiLocale` 作为当前母语建连

### 6.3 Suggest / Review / Assist

- `suggest`
  - 前端已传 `learningLanguage`
  - Edge Function prompt 已按学习语言生成
- `review`
  - 前端已传：
    - `learningLanguage`
    - `nativeLanguage`
  - Edge Function prompt 已按：
    - 解释类输出走母语
    - corrected / better expression 走学习语言
- `assist-reply`
  - 已按 `to_learning`
  - 输出目标语言为 `learningLanguage`

### 6.4 Onboarding

- 当前 onboarding 已感知 `learningLanguage`
- 会向用户显示当前正在学习的语言

## 7. 代码层约定

### 7.1 命名原则

后续新增语言相关业务字段时，优先使用：

- `nativeLanguage`
- `learningLanguage`

不要优先使用：

- `helpLocale`
- `assistantLanguage`
- `targetLocale`

除非场景确实需要更细粒度的能力拆分。

### 7.2 Prompt 入参原则

后续所有 LLM / translation / recommendation 链路，优先显式传：

```ts
{
  nativeLanguage: "...",
  learningLanguage: "..."
}
```

而不是只传一个模糊的 `locale`。

## 8. 当前仍保留的现实边界

虽然语言模型已经收敛为双语言模型，但当前工程里仍有这些现实边界：

- UI 文案当前只完整支持：
  - `en`
  - `zh-CN`
- `learningLanguage` 已经可以选择多种语言
- 但不代表所有实时 ASR / TTS / prompt / coach / lesson 能力都已经在每种学习语言下做过完整验证

也就是说：

- “模型已收敛”
- 不等于
- “所有学习语言能力都已完全成熟”

## 9. 后续实现建议

下一批优先建议：

1. 把 `learningLanguage` 继续接入 `coach` / `lesson` / 场景推荐流
2. 检查 TTS 在非英语学习语言下的表现
3. 检查主 websocket 在多学习语言下的 ASR 参数与 provider 支持
4. 如果未来真的需要第三种语言概念，再单独定义，不要提前过度抽象

## 10. 一句话原则

TalkPilot 当前最准确的语言原则是：

- `用户理解靠母语`
- `用户表达靠学习语言`

后续所有语言相关实现，都应围绕这条原则展开。
