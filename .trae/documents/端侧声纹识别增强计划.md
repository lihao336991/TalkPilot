# 端侧声纹识别增强计划

## Summary

在现有 `Deepgram diarization + enrollment priming` 的基础上，新增一层 **iOS 端侧声纹相似度判定**，用于增强“本人语音”的识别准确性与稳定性。

本期范围锁定为：

- `iOS first`
- `Hybrid gate`
- 保留现有 Deepgram speaker diarization 与 enrollment priming
- 新增本地 embedding / similarity / 状态机判定，用来对 `self` 进行优先确认和纠偏
- Android 本期不实现，仅保留接口抽象与降级路径

目标不是替换 ASR，而是提升这条链路的稳定性：

1. 更稳定地区分 `self` 与 `other`
2. 降低本人发言被误判成 `other` 的概率
3. 在 Deepgram diarization 抖动、speaker id 漂移或短句场景下提供本地纠偏

## Proposed Changes

### 变更 1：把 enrollment 从“保存 PCM”升级为“保存 PCM + embedding + 元数据”

#### 文件

- `src/features/live/services/VoiceEnrollmentService.ts`

#### 改动内容

扩展 enrollment 数据结构，新增一份持久化元数据，例如：

```ts
type VoiceEnrollmentProfile = {
  version: 1;
  createdAt: number;
  sampleRate: 16000;
  durationMs: number;
  embedding: number[];
  model: 'ecapa-coreml';
  thresholdSelfHigh: number;
  thresholdSelfLow: number;
};
```

保留当前 PCM 文件，新增一个 JSON profile 文件：

- PCM 继续用于：
  - Deepgram priming
  - 设置页试听
- embedding profile 用于：
  - 端侧实时相似度比较
  - 阈值与模型版本管理

#### 为什么这样做

- 不破坏当前已经稳定工作的 priming 能力
- 让端侧声纹与 Deepgram priming 共存
- 后续若模型升级，可基于 `version/model` 平滑失效旧 profile

#### 额外接口

在 `VoiceEnrollmentService` 中新增只读/辅助方法：

- `loadEnrollmentProfile()`
- `saveEnrollmentProfile(profile)`
- `hasEnrollmentProfile()`
- `clearEnrollmentProfile()`
- `loadEnrollmentPcmBase64()`

说明：

- 现有 `hasEnrollment()` 建议升级为同时校验 PCM 与 profile 是否齐备
- 如只有 PCM 没有 profile，应视为“旧格式 enrollment”，启动时引导重建

### 变更 2：新增 iOS 原生声纹模块，负责 embedding 推理

#### 文件

- `ios/TalkPilot/VoiceprintModule.swift`（新建）
- `ios/TalkPilot/VoiceprintModuleBridge.m`（新建）
- `src/features/live/services/VoiceprintNative.ts`（新建）

#### 模块职责

提供最小可用的原生接口，统一由 JS service 调用：

```ts
type NativeVoiceprintModule = {
  generateEmbedding(base64Pcm: string): Promise<number[]>;
  compareEmbedding(base64Pcm: string, enrollmentEmbedding: number[]): Promise<{
    similarity: number;
    embedding?: number[];
  }>;
  isAvailable(): Promise<boolean>;
};
```

#### 设计决策

- iOS first：原生实现只落在 `ios/`
- JS 层通过 `VoiceprintNative.ts` 做平台封装
- Android / Web 统一返回 `isAvailable = false`
- 推理能力不直接散落在业务 hook 中，集中到 service 层

#### 模型方案

本期计划按文档方向落地为：

- 采用 **ECAPA-TDNN 或同级 speaker embedding 模型**
- 打包为 iOS 可调用的本地模型资产
- 优先走 **CoreML 推理**

原因：

- 你当前主要以 Xcode/iOS 为主工作流
- CoreML 对 iOS 端部署更自然
- 相比“把 Python 方案直接搬进 App”，更适合客户端工程

注意：

- 计划中不把模型转换脚本写死进仓库主流程
- 模型产物建议落到类似 `ios/TalkPilot/Models/` 目录
- 若当前拿不到稳定 CoreML 模型，可退化为“先接入 native stub + JS 接口 + 打分链路”，模型资产后补；但本计划默认以可接入 CoreML 模型为目标

### 变更 3：新增 JS 侧声纹服务，管理打分、窗口聚合、状态机

#### 文件

- `src/features/live/services/VoiceprintService.ts`（新建）

#### 职责

统一管理以下逻辑：

1. enrollment embedding 生成
2. 实时 PCM chunk 聚合为声纹分析窗口
3. 调用原生模块生成当前窗口 embedding / similarity
4. 输出结构化的本人判定结果
5. 管理投票、防抖、动态更新

#### 关键输入输出

```ts
type VoiceprintDecision = {
  similarity: number | null;
  label: 'self' | 'other' | 'unknown';
  confidence: 'high' | 'medium' | 'low';
  reason:
    | 'similarity_high'
    | 'similarity_low'
    | 'between_thresholds'
    | 'insufficient_audio'
    | 'native_unavailable';
};
```

#### 实时窗口策略

按当前工程事实，音频 chunk 来自 `AudioEngine.start(onAudioData)` 的 base64 PCM 数据流，因此在 JS service 内做窗口聚合：

- 分析窗口：`1.5s`
- 步长：`0.5s`
- 采样格式：沿用现有 `16kHz mono 16-bit PCM`

这意味着：

- `sendAudioRef.current` 在把 chunk 送给 Deepgram 之前或同时，也要送给 `VoiceprintService.ingestChunk()`
- `VoiceprintService` 只在窗口满足最小时长时才触发一次本地比较

#### 阈值策略

本期先按固定阈值落地，后续再开放调参：

- `>= 0.80`：强判 `self`
- `<= 0.65`：强判 `other`
- 中间区间：`unknown`

之所以比参考文档更保守一些：

- 当前工程还没有线下评测集
- 需要优先降低“误把对方判成自己”的风险

#### 多帧投票与防抖

在 `VoiceprintService` 内维护最近 N 个窗口结果：

- 连续 `2` 个高阈值窗口命中，才把本地结论提升为 `self`
- 连续 `2` 个低阈值窗口，才把本地结论提升为 `other`
- 中间分数保持 `unknown`

再叠加一个简单状态机：

- 当前状态为 `self` 时，单次轻微跌破高阈值不立即翻转
- 当前状态为 `other` 时，需要连续高置信窗口才能切回 `self`

#### 动态更新

本期只做保守版动态更新：

- 仅在同时满足以下条件时更新 enrollment embedding
  - Deepgram 判定为 `self`
  - 端侧 similarity `>= 0.88`
  - 非 forced speaker
- 更新方式：

```ts
updated = normalize(0.9 * old + 0.1 * current)
```

这样做的目的：

- 避免把污染样本写入 enrollment
- 只在“高置信本人”场景下缓慢自适应

- 但不启用端侧声纹增强

### 变更 7：将 `DeepgramStreamingService.determineSpeaker()` 改为 Hybrid 决策

#### 文件

- `src/features/live/services/DeepgramStreamingService.ts`

#### 当前问题

当前判断路径基本是：

1. `forcedSpeaker` 最高优先级
2. 读 Deepgram `words[].speaker`
3. 若 `selfSpeakerId` 为空，自动锁定首次 majority 为 `self`

问题在于：

- Deepgram diarization 在短句、重叠、噪音、speaker id 漂移场景不够稳
- 一旦初次 auto-lock 锁错，会放大后续误判

#### 新的融合优先级

本期改为下面的决策顺序：

1. `forcedSpeaker` 最高优先级，保持不变
2. 如果端侧声纹强判 `self`，优先输出 `self`
3. 如果端侧声纹强判 `other`，且 Deepgram 也不是高置信 `self`，输出 `other`
4. 否则回退到 Deepgram diarization + `selfSpeakerId`
5. 若 Deepgram 仍不可靠，则使用上一个稳定 speaker 状态做短时延续

#### 具体规则

定义三个输入：

- `voiceprintDecision`: `self | other | unknown`
- `deepgramRawSpeakerId`
- `selfSpeakerId`

融合逻辑：

- `voiceprintDecision === 'self'`:
  - 直接输出 `self`
  - 若当前 `selfSpeakerId === null` 且 Deepgram 有 raw speaker，可把该 raw speaker 记录为候选 self id
- `voiceprintDecision === 'other'`:
  - 若 Deepgram majority 也不是 `selfSpeakerId`，输出 `other`
  - 若二者冲突，优先保持上一稳定状态，不立刻翻转
- `voiceprintDecision === 'unknown'`:
  - 完全回退到现有 Deepgram 逻辑

#### `selfSpeakerId` 的策略调整

当前：首次 majority 直接 auto-lock

调整为：

- 只有在以下条件满足时才允许 auto-lock：
  - 端侧声纹当前不是 `other`
  - enrollment profile 已就绪
  - 或者当前仍处于 priming 成功后的启动阶段

这样可以降低错误 speaker id 首次绑定的风险。

#### 输出调试信息

在 `DeepgramStreamingService` 内部日志与 debug trace 中增加：

- `voiceprintSimilarity`
- `voiceprintDecision`
- `speakerDecisionSource`

用于后续调参。

## Detailed Data Flow

### A. Enrollment

1. 用户在 `VoiceEnrollmentCard` 录 5 秒 PCM
2. `VoiceEnrollmentService.saveEnrollment()` 保存 PCM
3. `VoiceprintService.createEnrollmentProfileFromChunks()` 调原生模型生成 embedding
4. `VoiceEnrollmentService.saveEnrollmentProfile()` 持久化 embedding profile
5. 设置 `voiceprintEnrollmentReady = true`

### B. Session Start

1. `useLiveSessionController.handleStartSession()` 校验 enrollment
2. `startStreaming()` 先连接 Deepgram
3. 继续执行现有 enrollment priming，尽量锁定 `selfSpeakerId`
4. 同时 `VoiceprintService.loadEnrollmentProfile()` 载入本地 embedding
5. 开启音频后，实时 chunk 同时送往：
   - `deepgramService.sendAudio(base64)`
   - `voiceprintService.ingestChunk(base64)`

### C. Realtime Decision

1. `VoiceprintService` 每 0.5s 产出一次相似度与本地决策
2. `DeepgramStreamingService` 收到 Results / UtteranceEnd 时读取最新 voiceprint 决策
3. `determineSpeaker()` 使用 hybrid 规则生成最终 speaker
4. speaker 与文本继续走现有 turn 提交、翻译、suggest、review 流程

### D. Session End / Reset

1. `handleEnd()` 清空 voiceprint runtime state
2. 设置页重置 enrollment 时同时清除 PCM + profile
3. 若 profile 版本与当前模型不匹配，自动视为失效并提示重录

## Assumptions & Decisions

### 已确认决策

- 平台范围：`iOS first`
- 融合方式：`Hybrid gate`
- 不替换现有 Deepgram ASR
- 不移除现有 enrollment priming
- `forcedSpeaker` 继续保留最高优先级

### 本计划中的工程决策

- 端侧声纹只负责增强 `self` 判定，不单独产生转写文本
- 保留 PCM 与 embedding 双存储
- 不把 embedding 放进 zustand
- 阈值先静态写在 service/config 内，不在本期开放用户可调 UI
- 动态更新只做保守版，且必须经过双重高置信门控
- debug 信息只在现有 debug 能力上增强，不新增单独调试页

### 风险与对策

#### 风险 1：模型资产接入周期不可控

对策：

- 计划中把 native bridge、JS service、fusion state 先设计清楚
- 模型文件作为独立资产接入，不把业务逻辑与具体模型格式耦死

#### 风险 2：实时推理过于频繁导致卡顿

对策：

- 固定 1.5s 窗口、0.5s 步长
- 在 JS 层做节流与窗口复用
- 只在满足最小时长后触发 native compare

#### 风险 3：端侧声纹与 Deepgram 互相冲突，导致 speaker 抖动

对策：

- 不直接“谁高谁赢”，而是使用 Hybrid 状态机
- 端侧只在高置信时强介入
- 中间区间回退到现有 diarization

#### 风险 4：旧 enrollment 数据格式与新方案不兼容

对策：

- 通过 profile 文件版本区分
- 允许旧 PCM 继续用于 priming
- 但不给端侧增强能力，提示用户重录

## Acceptance Criteria

1. enrollment 完成后，除 PCM 外还能生成并保存一份本地 embedding profile
2. settings 可区分“仅旧样本”与“增强识别已激活”
3. 主链路实时录音时，音频同时进入 Deepgram 与端侧声纹分析
4. `forcedSpeaker` 仍保持最高优先级，不受新逻辑影响
5. Deepgram diarization 抖动时，端侧声纹可在高置信场景下维持 `self` 判定稳定
6. 端侧声纹不可用、模型失败或 enrollment profile 缺失时，链路自动降级回现有逻辑，不影响会话正常开始
7. pause / resume / assist / end 不会污染主声纹分析窗口
8. DebugOverlay 能看到最近 similarity、voiceprint decision、最终 speaker source

## Verification Steps

### 静态验证

1. `npx tsc --noEmit`
2. 检查新增 TS 文件无类型错误
3. 检查 i18n key 覆盖完整

### 手工验证

1. 首次无 enrollment 时开始会话，仍会弹出录音引导
2. 完成 enrollment 后，设置页状态显示“增强识别已启用”
3. 重置 enrollment 后，PCM 与 profile 同时清空
4. 正常进入会话时：
   - Deepgram priming 仍执行
   - 端侧声纹状态变为 ready
5. 用户本人连续说短句时，speaker 不应频繁抖到 `other`
6. 对方说话时，在高相似度误触发概率可控前提下，不应大面积误判为 `self`
7. 进入 assist 录音并取消，再回主链路，不应出现上一轮 voiceprint 窗口污染
8. enrollment profile 缺失或 native 不可用时，会自动降级，不阻塞主功能
