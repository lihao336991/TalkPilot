# TalkPilot 合作与调试经验沉淀

这里记录了我们在开发 TalkPilot 过程中踩过的坑、吸取的教训，以及为了提高合作效率而制定的原则。
AI 在每次完成复杂调试或发现新的认知盲区后，必须**自觉、主动**地将新经验补充到本文件中，实现自我迭代。

## 1. 故障排查与调试原则 (Debugging Principles)

### 1.1 信任用户的直接反馈，优先验证最小变量

- **坑与教训**：在排查“救场功能无法识别中文”时，用户已经明确指出“我的系统语言是中文，怎么会识别成en”。但我没有第一时间去验证“运行时传出的语言变量到底是什么”，反而凭直觉去推演录音链路、音频编码、网络上传等复杂假设，导致绕了一大圈，做了大量无效的修改。最后仅仅是把语言参数写死成 `zh-CN` 就解决了问题。
- **行动指南**：
  1. **用户是对的**：当用户给出明确的高置信度反馈时（例如“语言识别错位”），必须把这条线索作为**最高优先级**去排查。
  2. **证据闭环第一**：不要在没有看到“关键变量到底输出了什么”之前，就去动底层链路代码。先用 `console.log` 甚至 `Alert` 拿到最直接的证据（比如 `source_language` 发了什么）。
  3. **做最小验证**：在怀疑是大链路问题前，先做一个“一秒钟能改完”的验证（比如强制固定一个参数），而不是一上来就重构。

### 1.2 区分“主观期望状态”与“代码运行时状态”

- **坑与教训**：用户期望的系统语言是中文，但在 React Native 配合某些第三方库（如 `Intl`）在特定 iOS 环境下，取到的 `locale` 可能是默认的 `en`。这种“期望与实际的错位”很容易引发玄学 Bug。
- **iOS 本地化机制（深层原因）**：在原生 iOS 开发中，如果 App 的包（`Info.plist`）里没有显式声明它支持某种语言（比如中文），那么即使用户的手机系统设置成中文，iOS 也会强制让这个 App 回退到它默认支持的语言（即 `en`）。这就导致了系统是中文，但 JS 里取出的却是英文。
- **行动指南**：
  1. 永远不要理所当然地认为 `API 返回值 === 用户眼睛看到的状态`。
  2. 涉及到设备信息（系统语言、时区、网络状态、设备型号）时，必须以代码真实打出来的 Log 为准。
  3. **环境隔离**：在 Expo/RN 项目中，如果要准确获取设备系统语言，必须在 `app.json` 和 `Info.plist` 中配置 `"CFBundleAllowMixedLocalizations": true`。并且这类**原生层配置（如 Info.plist 的修改）必须重新编译整个原生应用（如在 Xcode 重新 Run 或者跑 `npx expo run:ios`）才能生效，不能仅仅依赖 JS bundle 的热更新。**

### 1.3 React Native 音频链路的特殊性

- **坑与教训**：
  1. **FormData 上传本地文件 URI 是巨坑**：在 RN 中用 `FormData` 上传 `file://` 路径的音频文件，很容易在后端收到空文件或损坏文件。更稳定的做法是：客户端转 Base64 字符串发 JSON，服务端再转回二进制 Buffer。
  2. **音频会话（AudioSession）冲突**：在 iOS 真机上，如果主通话引擎已经占用了一个 `VoiceChat` 模式的 AudioSession，你再用另一个库（比如 `expo-av`）去独立录音，可能录出来的就是有长度但没人声的静音文件。尽量统一整个 App 的底层采集管线。

### 1.4 先确认 transcript 的来源，再做接口职责收敛

- **坑与教训**：把后端接口从“接收音频并转写”改成“直接接收 transcript”时，不能想当然地把任何现有 transcript 源直接复用。TalkPilot 当前主通话实时 ASR 在 `DeepgramStreamingService` 里固定使用 `language=en`，这与“中文救场”输入不是同一能力。
- **行动指南**：
  1. 先分清这次改动是在做“接口职责收敛”还是“上游识别能力替换”，不要把两件事混在一起。
  2. 改成 transcript 入参前，必须先核对上游 transcript 的真实来源、语言和触发时机是否匹配目标场景。
  3. 如果上游 transcript 还未独立建设完成，客户端调用层要显式暴露 transcript 参数，不要在服务端继续偷偷兜底音频转写。

### 1.5 长连接 ASR 复用时，要区分“正常结束”与“用户取消”

- **坑与教训**：母语救场的 ASR 走的是长连接 WS。正常完成录音时，可以 `Finalize` 后继续短暂保活复用；但如果用户中途取消，上一轮尚未完全结算的 transcript 包仍可能在几百毫秒后回到客户端。如果这时直接复用同一条 WS 开启下一次 capture，这些旧包就会落入新的 `activeCapture`，表现为 transcript/缓冲污染下一次录音。
- **行动指南**：
  1. 取消录音时，不要只清本地状态；要同时中止当前 capture 对应的远端流状态。
  2. 对可复用的实时 ASR 连接，优先把“正常完成”与“取消”做成两条不同路径。
  3. 如果不能严格证明服务端缓冲已经清空，取消后就直接断开并在下次重新建连，不要赌复用。

### 1.6 Supabase Edge Functions 的 Deno 诊断要与 RN 工作区隔离

- **坑与教训**：TalkPilot 是 Expo/RN 主工程，但 `supabase/functions/` 下是 Deno Edge Functions。编辑器或单文件 TS 诊断若仍沿用根 `tsconfig.json` 的 Node/RN 语义，就会把远程 `https://` import、`.ts` 扩展导入和 `Deno.env` 统统报错，即使运行时代码本身没有问题。
- **行动指南**：
  1. 不要一看到 `找不到模块 https://...` 或 `找不到名称 Deno` 就去改业务逻辑，先确认是不是工作区把 Deno 文件按普通 TS 在分析。
  2. 对 `supabase/functions/` 单独维护 Deno 配置，例如 `.vscode/settings.json` 的 `deno.enablePaths` 与 `supabase/functions/deno.json`，同时把该目录从根 `tsconfig.json` 排除。
  3. 如果当前诊断工具或编辑器仍以“单文件普通 TS”模式报错，可补一个仅供编辑器使用的 shim 声明文件，并用 `/// <reference path="..." />` 显式引用；这种 shim 只为类型系统服务，不应改变运行时导入路径。

### 1.7 实时链路改造时，先抽“传输层”，再改“协议层”

- **坑与教训**：在把救场方案从“录音文件上传”切换到“双 WS”时，如果直接在 `DeepgramStreamingService` 和救场链路里各自实现连接、重连、保活，很快就会出现两套几乎重复但细节不一致的逻辑。这样后续一旦修 WebSocket 生命周期问题，就必须在两个 service 里各修一遍，风险很高。
- **行动指南**：
  1. 先把可复用的“建连、断连、自动重连、保活、发送、状态同步”抽到独立传输层，例如 `StreamingWebSocketClient`。
  2. provider service 只保留协议解析和业务副作用，例如 Deepgram 的 `Results`/`UtteranceEnd` 处理、turn 入库、debug 打点等。
  3. 做多链路并行时，状态命名必须从一开始就拆开，例如 `mainWsStatus` / `assistWsStatus`，不要等串台后再返工。

### 1.8 可拖拽悬浮按钮的手势命中判定，要用“当前绝对位置”，不要用旧的局部坐标基准

- **坑与教训**：悬浮按钮在被拖拽改过位置后，如果录音中的“是否滑出按钮区域”仍然用 `touchStart + dx/dy` 这种局部位移推算，命中区域很容易和按钮真实屏幕位置错位，表现出来就是取消态颜色闪烁、状态反复横跳。
- **行动指南**：
  1. 对可拖拽控件，命中判定优先使用当前屏幕绝对包围盒，而不是依赖旧的局部坐标基准。
  2. 如果一个控件同时支持“拖拽改位置”和“长按录音”，录音态与拖拽态要彻底分离，避免共用一套易漂移的坐标逻辑。
  3. 这类问题优先怀疑“坐标系不一致”或“位置更新后命中区域未同步”，不要先把锅甩给边界抖动或颜色样式优先级。

### 1.9 “长按录音，松手发送”不能继续依赖底层 ASR 的自动提交时机

- **坑与教训**：如果交互要求是“按住录音，松手发送，外部松手取消”，但底层实时转写链路仍然在 `UtteranceEnd` 时自动把 turn 提交出去，那么用户哪怕还没松手，只要中途停顿一下，文字也会被提前发出，导致取消形同虚设。
- **行动指南**：
  1. 这类交互要把“录音结束”和“消息提交”拆开处理，不能默认底层 ASR 的自动 `UtteranceEnd` 就等于“用户确认发送”。
  2. 在按住态下，允许 ASR 继续出 interim/final，但真正的 turn 提交时机必须由显式的 release/confirm 动作控制。
  3. 取消动作不仅要清 UI 状态，还要能丢弃本次按住期间已经积累的转写缓存，否则取消只是视觉取消，不是业务取消。

### 1.10 Expo 原生 Apple 登录接 Supabase 时，先警惕 nonce mismatch

- **坑与教训**：在 `expo-apple-authentication` + `supabase.auth.signInWithIdToken({ provider: 'apple' })` 的 native iOS 方案里，按常规做法把同一个 `nonce` 同时传给 Apple 和 Supabase，仍可能遇到 `invalid nonce` / `Nonces mismatch`。这类报错不一定是 Apple 配置错，更可能是当前链路里的 nonce 校验兼容问题。
- **行动指南**：
  1. 先把问题和 `audience` / `client id` 错误区分开，不要混在一起排查。
  2. 首轮联调目标是先跑通闭环时，可以优先做最小验证：临时移除 Apple native 登录链路里的 `nonce`。
  3. 在确认业务跑通后，再决定是等待上游修复，还是切换到 OAuth 流程重建更严格的 nonce 校验。

### 1.11 用 `typeof baseLocale` 做 i18n schema 时，要避免把“文案值字面量”错误地锁成类型

- **坑与教训**：第一版本地 i18n 接入时，直接把英文资源 `en` 用 `as const` 导出，再把 `typeof en` 当成其他语言包的 schema。结果 TS 会把每一个 value 推成精确字符串字面量，例如 `"Profile"`、`"Choose your plan"`，导致中文语言包全部报类型错误，看起来像“结构不匹配”，本质却是“值字面量被锁死”。
- **行动指南**：
  1. 需要“同构校验”时，不要直接把 `typeof en` 原样给其他语言实现使用。
  2. 应该先做一层递归映射，把所有叶子节点从字符串字面量放宽为 `string`，例如 `DeepStringify<typeof en>`。
  3. 判断 i18n 类型问题时，先分清是“key 结构不一致”还是“literal type 过窄”，避免误以为翻译文件本身写错了。

### 1.12 不要把所有 AI 输出语言抽象成一个统一的 `helpLocale`

- **坑与教训**：在多语言设置设计里，容易为了“概念统一”先抽一个 `helpLocale`，然后默认把所有 AI 生成内容都塞到这个维度里。但 TalkPilot 的 `suggest` 和 `review` 实际承担的是两种不同任务：`suggest` 是“给用户一个可直接说出口的目标语言回复”，`review` 则是“用用户当前界面语言解释问题，同时保留目标语言修正句”。如果把两者强行绑定到同一个输出语言，会很快违背真实学习体验。
- **行动指南**：
  1. 先按任务类型定义语言规则，再决定是否需要抽象层，不要先抽象再硬套场景。
  2. 对 TalkPilot 当前策略：
     - `suggest` 跟随 `Learning language`
     - `review` 的解释/表扬跟随 `App language`
     - `review` 的 corrected / better expression 保留 `Learning language`
  3. `helpLocale` 可以保留为未来扩展位，但不要在当前阶段把它包装成 `suggest/review` 的统一开关。

## 2. 合作与迭代约定 (Collaboration Rules)

- **避免过度推演**：遇到错误日志时，不要一口气给出 5 个大重构方案。先给 1 个最简单的验证方案。
- **保持环境纯净**：为了排查问题加了各种 log、存了各种中间文件，在确认修复后，必须负责任地清理干净，不要给项目留垃圾。
- **自觉更新经验**：这是我对自己下的硬性约束。以后每当：
  1. 踩了 RN 的新坑
  2. 绕了弯路被用户纠正
  3. 发现某个三方库的特殊用法
  - **必须主动**打开这个文件，把教训写下来。
